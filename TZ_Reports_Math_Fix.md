# Техническое задание: Улучшение математики в отчётах (aiAnalyst.js & ReportTrendChart.vue)

## Контекст и проблематика
В текущей реализации аналитики (backend `aiAnalyst.js` и frontend `ReportTrendChart.vue`) для расчёта дельты и процентов роста используется метод линейной регрессии (`_linregress`). Проценты считаются по концам регрессионной прямой: `pct = (fitLast - fitFirst) / |fitFirst| * 100`.

Это приводит к трём критическим проблемам:
1. **Абсурдные проценты (до +4872%)**: Если `fitFirst` (точка регрессии в начале периода) близка к нулю или отрицательна, деление на неё даёт гигантские, нереалистичные проценты.
2. **Искажение из-за неполных месяцев**: Текущий календарный месяц (например, 15 дней из 31) включается в регрессию как есть. Резкое падение абсолютного значения в конце ряда "ломает" тренд вниз.
3. **Несоответствие ожиданиям**: Клиенты ожидают сравнения реальных показателей (например, "Январь vs Май"), а не абстрактных точек аппроксимации.

## Цель
Сделать математику отчётов прозрачной, реалистичной и понятной для клиента, внедрив нормализацию неполных месяцев и разделив логику: регрессия — только для направления тренда (up/down), реальные значения — для процентов и абсолютных дельт.

## Задача 1: Нормализация неполного месяца (Backend & Frontend)
Необходимо экстраполировать данные текущего (неполного) месяца, чтобы они были сопоставимы с полными месяцами.

### Backend (`backend/src/services/reports/dataAggregator.js`)
В функции `_seriesMeta(rawSeries)` или там, где формируются `monthly_periods`:
1. Если месяц помечен как `is_partial` (текущий месяц), и в нём есть данные за `days` дней:
2. Рассчитать коэффициент нормализации: `normFactor = daysInMonth / days` (где `daysInMonth` — количество дней в этом месяце).
3. Умножить абсолютные метрики (`clicks`, `impressions`) на `normFactor`.
4. Метрики типа `ctr` и `position` (относительные) нормализовать **не нужно**.
5. Добавить флаг `is_normalized: true` к этому бакету, чтобы фронтенд знал, что данные экстраполированы.

*Важно:* Нормализацию применять только если прошло хотя бы 3-5 дней месяца, иначе экстраполяция будет слишком шумной. Если `days < 3`, лучше исключить этот месяц из расчёта трендов или оставить как есть, но не показывать процент.

### Frontend (`frontend/src/components/reports/ReportTrendChart.vue`)
1. При получении данных для графика, если последняя точка имеет флаг `is_normalized` (или если дата — текущий месяц), визуально выделить её (например, пунктирной линией или другим цветом маркера), чтобы клиент понимал, что это прогноз до конца месяца.
2. В функции `_linregress` использовать уже нормализованные данные.

## Задача 2: Изменение формулы расчёта дельты и процента (Backend & Frontend)

### Backend (`backend/src/services/reports/aiAnalyst.js`)
1. **Переписать `_regressAttr(input, key)`**:
   Вместо использования `fitFirst` и `fitLast` для расчёта дельты и процента, использовать **фактические (или нормализованные)** значения первой и последней точки ряда.
   ```javascript
   // Псевдокод новой логики
   const values = _seriesValues(input, key);
   const { slope, n } = _linregress(values); // Регрессия нужна только для dir (направления тренда)
   
   if (n < 2) return null;
   
   const firstVal = values[0];
   const lastVal = values[n - 1]; // Это значение уже должно быть нормализовано, если месяц неполный
   
   const delta = lastVal - firstVal;
   
   // Защита от деления на ноль или микро-базу
   let pct = null;
   let pct_unreliable = false;
   
   if (firstVal > 0) {
       pct = (delta / firstVal) * 100;
       if (firstVal < 10) { // Порог "микро-базы", можно настроить
           pct_unreliable = true;
       }
   }
   
   const dir = slope > 0 ? 'up' : (slope < 0 ? 'down' : 'stable');
   
   return { first: firstVal, last: lastVal, delta, pct, dir, pct_unreliable };
   ```

2. **Обновить `_canonicalNumbers` и `_fallbackSummary`**:
   Если `pct_unreliable === true` (или `firstVal` был 0), **не выводить процент** в текст отчёта и в дайджест, выводить только абсолютную дельту ("выросло на 50 кликов", а не "выросло на 5000%").

### Frontend (`frontend/src/components/reports/ReportTrendChart.vue`)
1. В `trendSummary` (строки ~260) аналогично изменить логику:
   ```javascript
   const firstVal = ds.data[0];
   const lastVal = ds.data[ds.data.length - 1]; // Нормализованное значение
   const delta = lastVal - firstVal;
   let pct = null;
   if (firstVal > 0) {
       pct = (delta / firstVal) * 100;
   }
   // Если pct слишком большой (например > 500%) из-за малой базы, 
   // можно опционально скрывать его в UI или показывать только дельту.
   ```
2. Убедиться, что линия тренда (`trendLines`) по-прежнему рисуется по `fitFirst` и `fitLast` (чтобы визуально линия была прямой регрессии), но цифры справа (в `trendSummary`) считались по реальным крайним точкам.

## Ожидаемый результат
1. Исчезнут абсурдные проценты (тысячи процентов при старте с нуля).
2. Текущий месяц не будет "обваливать" график и тренд, так как его значения будут пропорционально экстраполированы до конца месяца.
3. Текстовые выводы AI будут оперировать реальными (или прогнозируемыми нормализованными) цифрами первой и последней точки, что совпадает с логикой восприятия клиентов.
4. Полная синхронизация логики между бэкендом (генерация текста и дайджеста) и фронтендом (отображение цифр справа от графика).
