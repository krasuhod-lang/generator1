# GIST Checklist

Короткий operational checklist для быстрого анализа, сборки и аудита страницы по GIST-логике.

---

## 1. Проверка интента

- Понятно ли, какой у страницы основной интент?
- Есть ли вторичные интенты?
- Соответствует ли формат страницы запросу?
- Помогает ли страница решить конкретную задачу пользователя?
- Нет ли ухода в соседний интент?

Если нет четкого ответа хотя бы на 2 пункта — страницу нельзя считать готовой.

---

## 2. Проверка ядра темы

- Закрыто ли обязательное ядро темы?
- Есть ли у страницы все базовые элементы, которые пользователь реально ожидает?
- Не удалили ли важную базу в попытке быть «оригинальными»?
- Не раздут ли базовый материал сверх меры?

Правило:
- ядро должно быть сохранено,
- ядро не должно доминировать над остальной ценностью страницы.

---

## 3. Проверка смысловой избыточности

- Есть ли блоки, которые можно удалить без заметной потери пользы?
- Есть ли разделы, которые повторяют друг друга?
- Есть ли блоки, которые выглядят как типовой SERP-шаблон?
- Можно ли заменить часть страницы типовым текстом конкурента почти без потери смысла?
- Есть ли раздутые вступления, FAQ, списки или повторные объяснения?

Если да — страница избыточна.

---

## 4. Проверка недостающих смысловых узлов

- Есть ли исключения?
- Есть ли ограничения?
- Есть ли пограничные случаи?
- Есть ли ошибки и анти-паттерны?
- Есть ли случаи, когда типовой совет не работает?
- Есть ли скрытые критерии выбора?
- Есть ли реальные сценарии использования?
- Есть ли различия по типам пользователей, гео, устройствам или условиям?
- Есть ли цена ошибки?
- Есть ли доказательства, данные, наблюдения или редкие детали?

Если нет хотя бы части этих узлов там, где они релевантны теме, страница слишком стандартная.

---

## 5. Проверка заменимости

Спроси прямо:

- Можно ли заменить эту страницу одним из лидеров без большой потери пользы?
- Можно ли заменить первые 3–5 блоков страницы на типовой контент конкурента?
- Видна ли уникальная ценность уже в первой трети страницы?
- Есть ли блоки, которые реально труднее заменить?

Если страница легко заменима — она проваливается по GIST-логике.

---

## 6. Проверка первых блоков

- Сильные ли первые экраны?
- Есть ли ценность уже в начале, а не только дальше по тексту?
- Не начинается ли страница с общих слов?
- Не слишком ли поздно появляется главный differentiator?
- Видно ли уже в первых H2, чем эта страница полезнее стандартной?

Если лучший материал спрятан низко — подними его.

---

## 7. Проверка структуры

- У каждого раздела есть четкая функция?
- Нет ли нескольких разделов, которые делают одну и ту же работу?
- Логика движения по странице понятна?
- Нет ли разделов, добавленных только «для полноты»?
- Не выглядит ли структура как копия среднего SERP-шаблона?

Правило:
Структура должна быть полезной, а не просто знакомой.

---

## 8. Проверка полезности

Для каждого крупного блока спроси:

- Помогает ли он принять решение?
- Снижает ли он неопределенность?
- Добавляет ли он конкретику?
- Добавляет ли он критерии выбора?
- Добавляет ли он ограничения, исключения, сравнения, примеры или доказательства?

Если блок ни на что из этого не влияет — он слабый.

---

## 9. Проверка доказательности

- Есть ли в тексте конкретика, а не только уверенные формулировки?
- Есть ли примеры, кейсы, данные, сравнения, наблюдения?
- Есть ли места, где утверждение просит доказательства, но их нет?
- Есть ли «экспертный тон» без фактической опоры?

Если страница звучит уверенно, но не показывает опору, она уязвима.

---

## 10. Проверка FAQ

- FAQ реально нужен?
- Он добавляет новые узлы, а не повторяет тело текста?
- Вопросы не слишком очевидные?
- FAQ не выглядит как свалка хвостов?
- Его можно сократить без потери ценности?

Если FAQ шумит — сократить или удалить.

---

## 11. Проверка языка

- Нет ли воды?
- Нет ли длинных абстрактных вступлений?
- Нет ли повторов одной мысли разными словами?
- Нет ли фраз типа «это зависит от ваших потребностей» без конкретизации?
- Нет ли пустых подводок перед списками и таблицами?
- Нет ли безопасных, но ничего не значащих обобщений?

Язык должен быть плотным, ясным и прикладным.

---

## 12. Проверка действия по каждому блоку

Каждый блок должен получить одно из решений:

- Оставить.
- Оставить, но сократить.
- Переписать.
- Объединить с другим блоком.
- Поднять выше.
- Удалить.

Нельзя оставлять блок без решения, если его функция не ясна.

---

## 13. Быстрая таблица оценки

Используй для каждого крупного блока:

| Блок | Полезность | Риск избыточности | Отличимость | Решение |
|---|---|---|---|---|
| Вступление | Высокая / средняя / низкая | Низкий / средний / высокий / критический | Высокая / средняя / низкая | Оставить / сократить / переписать / удалить |
| H2 #1 |  |  |  |  |

---

## 14. Финальный вердикт

Перед публикацией выбери только один вариант:

- Готово к публикации.
- Готово после мелких правок.
- Нужна еще одна итерация.
- Текущая версия слишком слабая.

Если это не «Готово к публикации», обязательно назови причину:
- высокая заменимость,
- слабый differentiator,
- недостаток доказательств,
- структурная шаблонность,
- избыток воды,
- отсутствие важных смысловых узлов.

---

## Сверхкороткая формула GIST

Сильная страница = обязательное ядро + недостающие смысловые узлы + доказательства + низкая заменимость - смысловая избыточность.

Если страница длинная, гладкая и «вроде хорошая», но легко заменима лидером, она не проходит GIST-проверку.
