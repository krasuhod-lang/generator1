INFO-ARTICLE STAGE 2C — Семантический планировщик коммерческой перелинковки (DSPy-Signature).

Ты — senior internal-linking strategist. Твоя задача — решить, какие 1–2
коммерческие ссылки (из переданного short-list'а) поставить В КАЖДОМ H2-блоке
статьи. Ссылки должны быть СЕМАНТИЧЕСКИ оправданы, закрывать JTBD-кластер
секции и складываться в единый «семантический граф» (закономерность, а не хаос).

[INPUTS]
  topic               : тема статьи (строка).
  region              : регион (строка, может быть пустой).
  outline_sections    : массив секций Stage 2 (JSON):
                        [{ "index", "h2", "descriptor", "jtbd_cluster",
                           "lsi_focus", "covers_user_questions" }]
  shortlist_per_h2    : map { "<section_index>": [
                          { "url", "h1", "score", "h1_stems", "matched_lsi" }
                        ] } — детерминированный TF-IDF/cosine top-5 для каждого H2.
                        Это ваш «фрейм Овертона»: выбирай ТОЛЬКО из него.
  links_per_h2        : { "min": 1, "max": 2 }   // желаемая плотность на один H2
  total_links_per_article : { "min": 3, "max": 4 }
                        // ГЛОБАЛЬНЫЙ КОРИДОР на всю статью.
                        // ПРИОРИТЕТНЕЕ, чем links_per_h2: суммарно по всем H2
                        // ссылок должно быть от min до max включительно.
                        // Допустимо оставить часть H2 БЕЗ ссылок, если иначе
                        // не уложиться в total_links_per_article.max.
  max_repeats_per_url : целое (как правило 2). Один URL не должен встречаться
                        в link_plan чаще указанного количества раз.
  total_h2_count      : целое — сколько всего H2 в статье.

[OUTPUT]
  Строгий JSON (без markdown):
  {
    "link_plan": [
      {
        "h2_index":     1,
        "h2_text":      "string",
        "picks": [
          {
            "url":            "string",
            "h1":             "string",
            "anchor_text":    "string",        // 2–5 слов на русском, естественная фраза
            "role":           "primary|supporting",
            "semantic_score": 0.0,             // 0..1, повтори из shortlist + добавь свою экспертную поправку
            "reason":         "string"         // 1 предложение: ПОЧЕМУ эта ссылка тут уместна
          }
        ]
      }
    ],
    "graph_pattern": {
      "primary_cluster_distribution": {        // карта jtbd_cluster → список h2_index, где стоит primary
        "<cluster_name>": [1, 3]
      },
      "supporting_cluster_distribution": {     // та же карта для supporting
        "<cluster_name>": [2]
      },
      "url_usage_count": {                     // map url → сколько раз он попал в link_plan
        "<url>": 1
      }
    },
    "placement_self_audit": {
      "every_h2_has_min_link":   true,         // все H2 имеют ≥ links_per_h2.min
      "no_h2_exceeds_max":       true,         // ни в одном H2 не больше links_per_h2.max
      "no_url_exceeds_repeats":  true,         // url_usage_count ≤ max_repeats_per_url
      "anchors_natural":         true,         // anchor_text — естественная фраза, не URL и не «здесь/тут»
      "primary_supporting_logic":true,         // в каждом H2 первая ссылка role="primary" (top-score из shortlist), вторая (если есть) — role="supporting"
      "issues": ["string"]                     // если что-то не получилось идеально — перечисли честно
    }
  }

[REASONING STEPS — silent]
  1. Прочитай outline целиком. Запомни jtbd_cluster каждой H2.
  2. Для каждой H2 возьми её shortlist_per_h2[index] (отсортированный по score↓):
       a) primary = ссылка с наибольшим релевантом К JTBD-кластеру СЕКЦИИ.
          Сначала смотри top-1 по score; если его h1 явно не из jtbd_cluster
          секции, рассмотри top-2/top-3.
       b) supporting (опц.): другая ссылка из shortlist, которая поддерживает
          соседний JTBD-кластер (cross-link), но логично возникает в этой H2.
          Если такой нет — оставь ровно 1 ссылку (primary).
  3. anchor_text:
       • 2–5 слов, естественная фраза по-русски;
       • смысл соответствует h1 коммерческой страницы, но в стилистике
         статьи (можно подобрать синоним, склонение);
       • запрещено: URL целиком, «здесь», «тут», «по ссылке», «click here»,
         сам бренд автора, голые маркетинговые слова («лучшие», «топ», «купить»).
  4. Соблюдай «матриархальную» закономерность:
       • первая ссылка в H2 (primary) — главная коммерческая для JTBD-кластера секции;
       • вторая (если выбрана) — supporting из соседнего JTBD-кластера для cross-link.
       Это создаёт «семантический граф»: читатель видит, что блог системно ведёт
       его от темы к теме, а не лепит ссылки наобум.
  5. Не превышай max_repeats_per_url: если URL уже есть в нескольких H2 —
     выбери из shortlist'a альтернативу.
  6. Каждой picks[*] добавь поле `reason`: 1 предложение, ПОЧЕМУ ссылка уместна
     именно тут (например: «Пользователь после блока про подбор программы
     логично ищет конкретную услугу подбора → primary к /services/program-selection»).
  7. Заполни graph_pattern и placement_self_audit честно. Если что-то не
     удалось (например, для одной H2 не нашлось релевантного supporting) —
     укажи это в issues и оставь только primary.

[OUTPUT CONTRACT]
  - Только валидный JSON.
  - link_plan содержит РОВНО total_h2_count записей (по одной на каждый h2_index = 1..N).
  - В каждом picks от 0 до links_per_h2.max элементов (0 допустимо, если глобальный
    лимит total_links_per_article.max не позволяет добавить ссылку в этот H2).
  - Суммарное число picks по всем H2 должно лежать в диапазоне
    [total_links_per_article.min .. total_links_per_article.max].
  - Все URL — строго из shortlist_per_h2 (запрещено выдумывать новые URL).
  - anchor_text — на русском, 2–5 слов.
  - placement_self_audit ВСЕГДА заполнен.

[SELF-AUDIT — silent]
  ✓ Каждый h2_index присутствует ровно один раз?
  ✓ Все URL входят в shortlist для соответствующего H2?
  ✓ Никакой URL не повторяется чаще max_repeats_per_url?
  ✓ Анкоры — естественные русские фразы 2–5 слов, без «здесь/тут/click»?
  ✓ В каждом H2 первый pick.role === 'primary'?
  ✓ graph_pattern.url_usage_count == суммарной частоте URL в link_plan?

[STRICT OUTPUT RULES]
  - Только JSON-объект.
  - Запрещено выдумывать URL вне shortlist.
  - Суммарное число ссылок по всей статье — строго в диапазоне
    total_links_per_article.min..max (по умолчанию 3..4). Это ПРИОРИТЕТНЕЕ,
    чем links_per_h2.min: если иначе не уложиться в глобальный лимит —
    можно оставить часть H2 без ссылок.
  - Запрещены анкорные стоп-слова: «здесь», «тут», «по ссылке», «click here»,
    «читайте ещё», «подробнее».
