INFO-ARTICLE STAGE 2 — Структура статьи в блог: H1/H2/H3 + JTBD-теги + плейсхолдеры изображений (DSPy-Signature).

[INPUTS]
  topic               : тема статьи (строка).
  region              : регион (строка, может быть пустой).
  stage0_audience     : Stage 0 output (JSON).
  stage1_intents      : Stage 1 output (JSON) — entities, user_questions, jtbd_clusters.
  whitespace_hints    : Stage 1B → article_hierarchy_hints (JSON, обязательный).

[OUTPUT]

[REASONING STEPS — silent]
  1. Сформируй H1 (≤ 75 символов), отражающий тему, без маркетингового пафоса.
  2. Спланируй 6–10 H2 (включая «Заключение» и «Часто задаваемые вопросы»). Точное число подбирай по объёму темы.
  3. ⚠️ ЖЁСТКАЯ ТРАССИРУЕМОСТЬ СКЕЛЕТА. Каждый H2 (кроме «Заключения» и FAQ-блока)
     должен быть напрямую выведен из Stage 1: укажи в `derived_from`:
       • ≥1 entity из stage1_intents.entities[*].entity (точное совпадение строки),
       • ≥1 subintent из stage1_intents.subintents[*].intent,
       • ≥1 question из stage1_intents.user_questions[*].question.
     ЗАПРЕЩЕНО создавать H2, у которого нет якоря в этих трёх массивах.
     То же — для каждой H3: она должна детализировать одну из перечисленных
     entities или закрывать одну user_question.
  4. Каждой H2 присвой:
       • один jtbd_cluster (СТРОГО из stage1_intents.jtbd_clusters[*].cluster);
       • intent;
       • покрываемые user_questions и must_cover_subtopics;
       • lsi_focus (3–6 терминов из stage1.semantic_anchors / entities);
       • рекомендуемый объём в словах (120–280).
  5. Добавь H3 туда, где сабтопик распадается на 2–4 шага/различия.
  6. ⚠️ EXPERT_OPINION_SLOT (обязательно ровно 1).
     Выбери `target_section_index` — секция средней-поздней части статьи
     (НЕ первая, НЕ «Заключение», НЕ FAQ), куда естественно вписать
     «мнение эксперта». Заполни `expert_role` (без выдуманного ФИО),
     `focus`, `key_insight` (контр-интуитивное, опытное наблюдение),
     `tied_to_entities` (минимум 1 сущность из Stage 1).
  7. ⚠️ FAQ_BLOCK (обязательно).
     • Возьми 4–6 вопросов из stage1_intents.user_questions, ещё не закрытых
       целиком в основных секциях, ИЛИ те, которые требуют короткого
       прямого ответа («Сколько …?», «Можно ли …?», «Сколько стоит …?»,
       «Сколько занимает …?», «Что выбрать: A или B?»).
     • `place_after_section_index` = index последней содержательной секции;
       FAQ ставится между ней и «Заключением».
     • Каждый item.question — переформулировка, item.source_question —
       оригинал из Stage 1, item.tied_to_entities — ≥1 entity.
  8. image_plan: РОВНО 1 слот — это cover-изображение для публикации статьи
     на сайт (как hero / обложка). target_section_index — index одной из
     основных содержательных секций (НЕ FAQ, НЕ «Заключение»), концептуально
     отражающей главный визуальный посыл статьи. Изображение НЕ встраивается
     в HTML-текст статьи — оно отдаётся отдельным файлом, поэтому планируй
     один сильный универсальный визуал, а не серию иллюстраций по разделам.
  9. ⚠️ ВАЖНО: НЕ планируй вставку коммерческих ссылок здесь —
     это сделает Stage 2c (semantic link planner) на основе jtbd_cluster.
     Твоя задача — выдать чистый план «о чём статья», не «куда ссылаться».
 10. recommended_word_count помогает писателю выдержать целевую глубину.

[OUTPUT CONTRACT]
  - Только валидный JSON.
  - Русский язык в содержательных полях.
  - 6–10 sections, ровно 1 слот в image_plan (cover-обложка).
  - Каждая секция имеет НЕ ПУСТЫЕ jtbd_cluster, lsi_focus, covers_user_questions.
  - Каждая содержательная секция имеет НЕ ПУСТЫЙ derived_from.entities/subintents/questions
    (исключения — служебные секции «Заключение» и сама FAQ-секция).
  - target_section_index в image_plan — это `index` секции (НЕ позиция в массиве).
  - expert_opinion_slot — ровно один объект, target_section_index указывает
    на существующий `index` основной (не служебной) секции.
  - faq_block.items — массив длиной 4–6.

[SELF-AUDIT — silent]
  ✓ Каждый user_question из Stage 1 закрыт ровно одной H2/H3 ИЛИ вошёл в faq_block?
  ✓ Каждый must_cover_subtopic присутствует в covers_subtopics хотя бы одной секции?
  ✓ Каждая основная секция имеет derived_from с ≥1 entity, ≥1 subintent, ≥1 question?
  ✓ Каждая секция получила корректный jtbd_cluster из доступного списка?
  ✓ image_plan содержит ровно 1 cover-слот, привязанный к основной (не служебной) секции?
  ✓ expert_opinion_slot заполнен и target_section_index указывает на основную секцию?
  ✓ faq_block.items содержит 4–6 элементов; каждый item.source_question есть в stage1.user_questions?
  ✓ H1 не содержит CTA-маркетинга («лучший», «топ», «купить»)?

[STRICT OUTPUT RULES]
  - Только JSON.
  - Запрещены пустые sections / jtbd_cluster / lsi_focus.
  - Запрещён jtbd_cluster, не входящий в stage1.jtbd_clusters.
  - Запрещены H2/H3, не выводимые из Stage 1 (entities + subintents + user_questions).
  - Запрещено пропускать expert_opinion_slot или faq_block — оба обязательны.
