INFO-ARTICLE STAGE 0 — Аудитория и тон для информационной статьи в блог (DSPy-Signature).

[INPUTS]
  topic           : тема статьи (строка).
  region          : регион / гео (строка, может быть пустой).
  brand_facts     : короткие факты о бренде/нише (строка, может быть пустой).
  strategy_digest : компактная выжимка Pre-Stage 0 (JSON).

[OUTPUT]
  Строгий JSON (без markdown):
  {
    "audience_personas": [
      { "name": "string", "context": "string", "pains": ["string"],
        "barriers": ["string"], "motivations": ["string"], "reading_level": "beginner|intermediate|expert" }
    ],
    "tone_of_voice": {
      "register":       "formal|neutral|friendly|expert-casual",
      "sentence_length":"short|medium|long",
      "do_use":         ["string"],
      "avoid":          ["string"]
    },
    "pain_points":      ["string"],
    "entry_barriers":   ["string"],
    "content_voice":    "string"
  }

[REASONING STEPS — silent]
  1. Используй `topic`, `region`, `strategy_digest` как якорь. Не выходи за границы темы.
  2. Сформулируй 2–3 персоны блогового читателя (информационный интент!),
     каждую — с конкретным контекстом: кто он, что он ищет, где находится в
     момент чтения статьи (ещё до покупки).
  3. Для каждой персоны перечисли реальные боли и барьеры, которые статья обязана снять.
  4. Тон — экспертный, но дружелюбный (статья В БЛОГ, не реклама), с уважением к читателю.
     Если region задан — учти разговорные нюансы (например, для РФ — без англицизмов,
     где есть русский эквивалент).
  5. pain_points → перечень «вопросов читателя», которые лягут в H2.

[OUTPUT CONTRACT]
  - Только валидный JSON, без комментариев.
  - Русский язык для содержательных полей.
  - 2–3 персоны, 4–8 болей, 2–5 барьеров.
  - Длина строк ≤ 280 символов.

[SELF-AUDIT — silent]
  ✓ Нет ли выдуманной статистики, цитат, имён реальных людей, брендов?
  ✓ Все персоны разные (не дубли одной и той же)?
  ✓ Тон согласуется с темой (нишевый экспертный блог, а не «инфобизнес»)?
  ✓ pain_points напрямую выводимы из `topic` + `strategy_digest`?

[STRICT OUTPUT RULES]
  - Верни ТОЛЬКО JSON. Ничего снаружи.
  - Нельзя возвращать пустые массивы для `audience_personas` и `pain_points`.
  - Нельзя ссылаться на несуществующие факты, исследования, проценты.
