LINK-ARTICLE STAGE 3 — Написание ссылочной статьи на Gemini 3.1 Pro Preview (DSPy-Signature, E-E-A-T-aware).

[INPUTS]
  topic            : тема статьи (строка).
  anchor_text      : анкор, который ОБЯЗАН быть в тексте ровно 1 раз (строка).
  anchor_url       : URL ссылки (строка, валидный http/https).
  focus_notes      : на что делаем упор (строка).
  output_format    : 'html' | 'formatted_text' (всегда генерируем HTML; frontend конвертирует).

  // ── Аналитический контекст ──────────────────────────────────────
  // Если задействован LINK-ARTICLE KNOWLEDGE BASE (LAKB) как
  // systemInstruction (Gemini cachedContents), все ниже-перечисленные блоки
  // уже доступны как фон. В user-prompt они приходят в виде коротких
  // ссылок-указателей "[См. LAKB → §N]". Если LAKB недоступен — приходят
  // полные JSON-выжимки этих блоков:
  //   - stage0_audience      (персоны, тон)
  //   - stage1_intents       (entities, subintents, user_questions, semantic_anchors)
  //   - whitespace_hints     (article_hierarchy_hints из Stage 1B)
  //   - stage2_structure     (план секций, anchor_plan, image_plan)
  //   - prior_eeat_issues    (опционально, при corrective re-run): список issues
  //                          из Stage 5 E-E-A-T audit с fix_instruction'ами.

[OUTPUT]
  Строгий JSON (без markdown-обёрток):
  {
    "article_html":       "string",
    "self_audit": {
      "anchor_placed":                true,
      "anchor_section_index":         1,
      "anchor_within_first_20pct":    true,
      "all_user_questions_covered":   true,
      "all_subtopics_covered":        true,
      "image_slots_present":          [1, 2, 3],
      "hallucination_free":           true,
      "eeat_self_score":              8.0,         // 0..10, твоя честная самооценка
      "eeat_signals_used":            ["string"],  // какие E-E-A-T сигналы реально вшиты
      "notes":                        "string"
    }
  }

[REASONING STEPS — silent]
  1. Сначала мысленно прочитай весь план (stage2_structure) — H1, все H2/H3, anchor_plan,
     image_plan — и сверь с user_questions из Stage 1 + must_cover_subtopics из
     whitespace_hints. Убедись, что ни один вопрос читателя и ни один обязательный
     сабтопик не останутся без ответа.
  2. Если задан `prior_eeat_issues` — это значит писатель идёт на CORRECTIVE PASS.
     Прочитай каждую issue, сопоставь с местом в статье (поле `where`) и при
     написании выполни КАЖДЫЙ fix_instruction. Severity:
       - blocker: исправь полностью, без полумер;
       - major: устрани и убедись, что итоговая цена изменения не сломала
         ничего полезного из первого прохода;
       - minor: улучшение «по дороге».
  3. Пиши статью СЕКЦИЮ ЗА СЕКЦИЕЙ, строго в заданном порядке. Сохраняй перечень H2 и H3.
  4. Во ВСТУПЛЕНИИ сразу обозначь пользу материала. Не лей воду. 2–4 абзаца.
  5. В секции с anchor_plan.target_section_index (H2 #1 или #2) органично упомяни
     `anchor_text` ВНУТРИ связного предложения-мысли и оформи как:
         <a href="ANCHOR_URL">ANCHOR_TEXT</a>
     — ровно один раз во всей статье, именно внутри этой секции.
  6. В тех секциях, где stage2_structure.image_plan назначил слот, вставь на своё
     место строчку-плейсхолдер:
         <!-- IMAGE_SLOT_1 -->
         <!-- IMAGE_SLOT_2 -->
         <!-- IMAGE_SLOT_3 -->
     — каждый комментарий строго один раз, ровно в той секции, на которую
     указывает stage2_structure.image_plan[*].target_section_index, между абзацами.
  7. Следи за плотностью смысла: каждый абзац отвечает на конкретный вопрос
     читателя из user_questions. Старайся комбинировать теорию (что такое),
     практику (как), сравнение (чем отличается), ошибки (чего избегать).

[E-E-A-T REQUIREMENTS — обязательны]
  Статья оценивается по фреймворку E-E-A-T. Минимально допустимый суммарный
  балл (см. backend/src/utils/objectiveMetrics.js → EEAT_PQ_TARGET) = 7.5/10.
  Чтобы пройти аудит:

  • EXPERIENCE (опыт): встраивай практические наблюдения «с земли» —
    типичные ошибки, нюансы реального применения, неочевидные пограничные
    случаи. Такие фразы как «на практике это часто упирается в …»,
    «в реальных проектах чаще встречается …», «опытные специалисты обычно
    обращают внимание на …» — допустимы и полезны (это ОБОБЩЕНИЕ опыта,
    а не выдуманная цитата конкретного человека).

  • EXPERTISE (экспертиза): корректная терминология, точные определения,
    правильные различия между близкими понятиями. Если уместно — давай
    короткие чек-листы и пошаговые алгоритмы.

  • AUTHORITATIVENESS: чёткая структура, логичные переходы, отсутствие воды.
    Используй элементы экспертного контента: списки критериев, сравнения
    подходов, типовые сценарии, контр-примеры.

  • TRUSTWORTHINESS: НИКАКИХ выдуманных фактов. Запрещены формулировки:
       – «по данным исследования X», «согласно отчёту Y»;
       – «в 2023 году рынок вырос на N %»;
       – конкретные имена/цитаты реальных людей;
       – конкретные бренды, продукты, ISO-номера, регуляторные акты, если
         не уверен в точной формулировке. Если число нужно — переводи в
         обобщение («в большинстве случаев», «в ряде методов»).

  • БАЛАНС: показывай не только pros, но и cons / ограничения / условия,
    когда подход не работает. Это сильно повышает trust-сигнал.

[ARTICLE HTML FORMAT]
  - Статья — чистый HTML-фрагмент (без <html>, <head>, <body>).
  - Структура:
      <h1>…</h1>
      <p>…вступление (2–4 абзаца)…</p>
      <h2>Секция 1</h2>
      <p>…</p>
      <h3>…</h3>   (если нужно)
      <p>…</p>
      <!-- IMAGE_SLOT_N -->   (если в секции назначен слот)
      … и т.д.
      <h2>Заключение</h2>
      <p>…</p>
  - Разрешённые теги: h1, h2, h3, p, ul, ol, li, strong, em, a, blockquote.
  - Запрещено: <script>, <style>, <iframe>, inline onclick/onerror/onload.
  - Для значений атрибута href запрещены data:-URL (разрешены только http/https).
    При этом внутри статьи НЕ нужно вручную вставлять <img> — изображения
    добавляются на бэкенде путём замены плейсхолдеров <!-- IMAGE_SLOT_N -->
    на <figure><img src="data:image/..."><figcaption/></figure>. Это допустимое
    исключение для src-атрибута img и не противоречит запрету data: для href.
  - Все внутренние ссылки (кроме anchor) — только https:// и только если это
    общеизвестные официальные ресурсы. Если сомневаешься — не ставь ссылку.
  - Длина статьи: 1100–2000 слов (подбирай по глубине темы; для E-E-A-T нужен
    запас глубины относительно стандартных 900-словесных рерайтов).
  - Заголовок <h1> ровно один, он соответствует stage2_structure.h1.

[OUTPUT CONTRACT]
  - Только валидный JSON: `{"article_html": "...", "self_audit": {...}}`.
  - article_html — одна HTML-строка.
  - self_audit.anchor_placed === true и self_audit.image_slots_present равен [1,2,3].
  - self_audit.eeat_self_score — число в [0, 10] с одной цифрой после запятой.
  - Никаких markdown, backtick-обёрток, комментариев снаружи JSON.

[SELF-AUDIT — silent, но ОБЯЗАТЕЛЬНО заполни self_audit в выходе]
  ✓ anchor_text встречается ровно 1 раз внутри <a href="anchor_url">…</a>.
  ✓ Эта <a> находится в H2 №1 или H2 №2 (первые 20 % объёма текста).
  ✓ Все user_questions из Stage 1 закрыты.
  ✓ Все must_cover_subtopics из whitespace_hints разложены по секциям.
  ✓ Плейсхолдеры <!-- IMAGE_SLOT_1 -->, <!-- IMAGE_SLOT_2 -->, <!-- IMAGE_SLOT_3 -->
    присутствуют каждый ровно 1 раз.
  ✓ Нет выдуманных фактов, статистики, брендов, цитат реальных людей.
  ✓ Если задан prior_eeat_issues — каждая issue с severity ∈ {blocker, major}
    закрыта (отметь это в notes).
  ✓ HTML валиден: все теги закрыты, вложенность корректна.

[STRICT OUTPUT RULES]
  - Верни ТОЛЬКО JSON. Никаких префиксов/суффиксов.
  - Запрещено возвращать пустой article_html или заглушку.
  - Запрещено пропустить anchor или image-слоты.
  - При нарушении любого правила пересобери статью до тех пор, пока правило не выполнено.
