# Инструкция по обновлению — SEO Genius

Этот документ описывает **универсальную и безопасную** процедуру обновления
работающего инстанса SEO Genius на VPS:

1. Как обновить файл `.env` так, чтобы **никакие существующие данные не пропали**, а добавились только новые ключи.
2. Как корректно накатить обновление кода программы (любой релиз — `infoArticle`, `articleTopics`, `linkArticle`, `metaTags`, AI-Copilot и т.д.).

> ⚠️ **Главное правило:** `.env` **никогда не обнуляется и не пересоздаётся
> с нуля**. Все ваши API-ключи, прокси, пароли и кастомные настройки остаются
> на месте. Из релиза в релиз вы только **дописываете** новые опциональные
> переменные (если они вообще появились — почти всегда у них есть разумный
> default и можно вообще ничего не менять).

---

## TL;DR — короткая последовательность

```bash
ssh root@<your-server-ip>
cd /opt/seo-genius                    # путь до вашего проекта

# 1) Бэкап env (всегда!)
cp .env .env.backup.$(date +%Y%m%d_%H%M%S)

# 2) Подтянуть свежий код
git fetch --all
git status                            # убедиться, что локальных правок нет
git pull origin main                  # или имя вашей ветки

# 3) Сравнить .env c новым .env.example и дописать только НОВЫЕ ключи
diff <(grep -oE '^[A-Z_][A-Z0-9_]*' .env | sort -u) \
     <(grep -oE '^[A-Z_][A-Z0-9_]*' .env.example | sort -u)
#   строки с ">" = ключи, которых у вас нет → добавьте их в КОНЕЦ .env
nano .env                             # допишите недостающие ключи (см. §1)

# 4) Перезапустить контейнеры (миграции применятся идемпотентно при старте backend)
docker compose up -d --build backend frontend worker

# 5) Smoke-проверка
docker compose logs --tail=120 backend | grep -E 'Schema|Server|listening'
docker compose ps
```

Готово. Дальше — детали.

## §1. Безопасное обновление `.env`

### 1.1. Принцип «только дополняем»

`.env` — единственное место, где живут ваши секреты (`GEMINI_API_KEY`,
`DEEPSEEK_API_KEY`, `XAI_API_KEY`, прокси, пароли БД, JWT-секреты и т.д.).
Файл лежит в `.gitignore` и **никогда не приезжает в `git pull`** — значит,
обновление кода физически не может его испортить. Но **вы сами**, открыв
`nano .env`, можете случайно его перезаписать. Поэтому:

| Делаем | Не делаем |
|---|---|
| ✅ `cp .env .env.backup.$(date +%s)` перед каждым обновлением | ❌ Копировать `.env.example` поверх `.env` |
| ✅ Сравнивать ключи с `.env.example` через `diff` и **дописывать недостающие** в конец | ❌ Удалять из `.env` ключи, которых уже нет в `.env.example` (старые env могут ещё читаться кодом и нужны для бэк-совместимости) |
| ✅ Менять значение существующего ключа на новое только осознанно | ❌ Полностью пересоздавать файл с нуля |

### 1.2. Пошагово: «дописать только новое»

1. **Снимаем бэкап** (две команды на выбор):
   ```bash
   cp .env .env.backup.$(date +%Y%m%d_%H%M%S)
   # или с понятной меткой релиза:
   cp .env .env.backup.before-update-2026-04
   ```
   Бэкапы лежат рядом с `.env`, никуда не уезжают, и в любой момент можно
   откатиться: `cp .env.backup.<метка> .env && docker compose restart`.

2. **Сравниваем список ключей** (значения нам не интересны):
   ```bash
   diff <(grep -oE '^[A-Z_][A-Z0-9_]*' .env         | sort -u) \
        <(grep -oE '^[A-Z_][A-Z0-9_]*' .env.example | sort -u)
   ```
   - Строки с `>` — ключи **есть в `.env.example`, но нет в вашем `.env`**.
     Это и есть «новые опциональные переменные»; их нужно дописать.
   - Строки с `<` — ключи, которые есть **только** у вас. Не трогаем — это
     ваши кастомизации либо устаревшие, но всё ещё читаемые переменные.
   - Если `diff` ничего не вывел — обновлять `.env` не нужно вообще.

3. **Дописываем недостающие ключи в конец `.env`**. Открываем редактор:
   ```bash
   nano .env
   ```
   В самый низ добавляем блок (пример из недавних релизов — у вас могут быть
   другие ключи; смотрите `.env.example` как источник правды):
   ```env
   # ── Добавлено при обновлении 2026-04 ─────────────────────────────
   # (необязательно: у каждой переменной есть рабочий default)

   # GEMINI_TIMEOUT_MS=480000        # 8 минут на reasoning-ответ writer
   # STAGE8_EVALUATOR_ENABLED=false  # LLM-судья после Stage 7
   # INFO_ARTICLE_GEMINI_CACHE_ENABLED=true
   ```
   Все строки можно оставить **закомментированными** — тогда сработают
   дефолты из кода. Раскомментируйте только те, что хотите явно переопределить.

4. **Проверяем синтаксис** (docker compose сам подскажет, если что-то сломано):
   ```bash
   docker compose config --quiet && echo "✅ env корректен"
   ```

5. **Перезапускаем сервисы** (см. §2.4) — без `--build`, если код не менялся;
   с `--build`, если только что был `git pull`.

### 1.3. Чего НЕ делать

* ❌ `mv .env.example .env` — затрёт все ваши ключи.
* ❌ `cat .env.example >> .env` — задублирует ключи (последнее значение
  победит, но дубликаты усложнят отладку).
* ❌ Удалять из `.env` «лишние» ключи, которых нет в `.env.example`. В коде
  могут оставаться fallback-чтения (например, `LLM_PROXY_*` читается даже
  если в `.env.example` теперь только `GEMINI_PROXY_*`).
* ❌ Коммитить `.env` в git. Файл уже в `.gitignore`; убедитесь
  командой `git status`, что `.env` **не показан** ни в staged, ни в untracked.

### 1.4. Откат

Если после обновления что-то сломалось:
```bash
cp .env.backup.<нужная_метка> .env
docker compose restart backend worker
```

## §2. Внесение изменений в программу

Программа состоит из 4 docker-сервисов: `backend`, `worker`, `frontend`,
`postgres` (плюс опционально `redis`). Обновляются они одной командой
`docker compose up -d --build`. Миграции БД идемпотентны и применяются
автоматически при старте `backend` (`ensureSchema()`), поэтому вручную
ничего запускать не нужно.

### 2.1. Подключиться к серверу и обновить код

```bash
ssh root@<your-server-ip>
cd /opt/seo-genius                    # ваш путь установки

git fetch --all
git status                            # должно быть «working tree clean»
git pull origin main                  # либо имя вашей ветки
```

Если у вас НЕ git-деплой, а ручная заливка — перезалейте каталоги
`backend/`, `frontend/`, `migrations/` и обновлённый `.env.example`
(трогать `.env` не нужно — см. §1).

### 2.2. Обновить `.env`

Выполните §1.2 (бэкап → diff → дописать новые ключи → проверить).
Если `diff` пуст — переходите к 2.3.

### 2.3. (опционально) Накатить миграции вручную

При старте контейнера `backend` функция `ensureSchema()` сама применяет
все миграции из `migrations/*.sql` идемпотентно (каждая обёрнута в
`CREATE TABLE IF NOT EXISTS` / `ALTER TABLE … ADD COLUMN IF NOT EXISTS`).
Делать что-то руками не нужно. Но если хочется проверить — после
перезапуска backend в логах будет:
```
[Schema] applied 0XX_*.sql
[Schema] OK (NN tables ensured)
```

### 2.4. Пересборка и перезапуск контейнеров

```bash
# Полный путь — если менялся и backend, и frontend, и worker:
docker compose up -d --build backend frontend worker

# Только backend (например, фикс серверной логики):
docker compose up -d --build backend worker

# Только frontend (например, правка UI Vue):
docker compose up -d --build frontend
```

Флаг `-d` запускает в фоне. Флаг `--build` пересобирает образ из
свежего исходника. БД (`postgres`) не трогаем — её перезапускать не нужно.

### 2.5. Проверка работоспособности (smoke)

```bash
# Контейнеры подняты и здоровы?
docker compose ps

# В логах нет ошибок схемы?
docker compose logs --tail=120 backend | grep -E 'Schema|Server|listening|error'

# UI открывается? (замените на ваш домен/порт)
curl -sf http://localhost:3000/ >/dev/null && echo "✅ frontend OK"
curl -sf http://localhost:4000/api/health >/dev/null && echo "✅ backend OK"
```

Откройте веб-интерфейс и пройдитесь по разделам, которые задевал релиз.
Минимальный smoke на ключевые пайплайны:

| Раздел UI | Что проверить |
|---|---|
| **Темы статей** (`/article-topics`) | Создать main-задачу → дождаться `done` → открыть deep-dive → нажать «📝 Создать статью для блога» (на тренде или в «Быстрой победе»). Должен открыться `/info-article` с предзаполненными `Тема статьи` / `Регион` / `brand_facts`. |
| **Статья для блога** (`/info-article`) — БЕЗ Excel | Заполнить только `Тема` + `Регион` (Excel не загружать), нажать «Создать». В логах должно появиться `🚫 Excel-база не загружена → статья будет сгенерирована БЕЗ перелинковки`. Готовая статья НЕ должна содержать коммерческих `<a>`. |
| **Статья для блога** — С Excel | Загрузить Excel с колонками `url` + `h1` (или `ссылка` + `название`), создать задачу. В готовой статье на каждом `<h2>` должно быть 1–2 коммерческих ссылки из вашей базы. |
| **AI-Copilot** (`/copilot-hub`) | Открыть готовую статью, прогнать любое действие (например, «Добавить FAQ») — стрим должен идти, текст применяться. |

Если что-то упало — смотрите подробные логи:
```bash
docker compose logs --tail=400 backend
docker compose logs --tail=400 worker
docker compose logs --tail=200 frontend
```

### 2.6. Откат релиза

Если новая версия сломала что-то в проде:
```bash
# 1) Откатить код
git log --oneline -5                    # увидеть SHA предыдущего стабильного коммита
git checkout <previous-stable-sha>

# 2) (если меняли .env) восстановить старый
cp .env.backup.<метка> .env

# 3) Пересобрать
docker compose up -d --build backend frontend worker
```

Миграции БД написаны идемпотентно и **only-add** (никаких `DROP COLUMN` /
`ALTER … TYPE`), поэтому откат кода не требует отката схемы — старая
версия backend просто игнорирует новые колонки.

## §3. Что нового в текущем релизе («Статья для блога» + «Темы статей»)

### 3.1. Excel-база коммерческих страниц теперь **опциональна**

Раздел **«Статья для блога»** (`/info-article`) принимает задачи и **без**
загруженного Excel-файла. В этом случае:

* `Stage 2C Semantic Link Planner` — пропускается (link_plan = []);
* `Stage 5b Link Audit` — пропускается;
* writer (Stage 3 Gemini) получает маркер `[NO_INTERLINKING_MODE]` и **не вставляет коммерческих `<a href>`**;
* всё остальное (Stage 0/1 аналитика, LSI-набор, мнение эксперта, FAQ, image_slots, E-E-A-T аудит) — работает в полном объёме.

Если Excel **загружен**, но все ссылки отбракованы валидатором — задача
**отклоняется с 400** (как и раньше): значит, файл реально битый и
пользователь должен либо его исправить, либо отправить задачу без файла.

Никаких миграций БД для этого режима не требуется.

### 3.2. Из «Темы статей» (Phase 2 / deep-dive) генерация статей теперь идёт через «Статью для блога»

Кнопки **«📝 Статья для блога»** (на карточке тренда) и
**«📝 Создать статью для блога →»** (под блоком «Быстрая победа»)
теперь открывают `/info-article` с уже заполненными:

* `Тема статьи` ← `Ниша — Тренд` (или `Ниша — Trend name`);
* `Регион` ← регион из main-задачи;
* `brand_facts` ← драйверы / вектор / сигналы тренда (или весь блок «Быстрой победы»).

Раньше эти кнопки уводили в `/tasks/new` (классический SEO-пайплайн на
коммерческую страницу), что не соответствовало foresight-задаче «написать
блог-статью под выявленный тренд».

Excel-базу при этом загружать **не нужно** — автоматически срабатывает
режим «без перелинковки» (см. §3.1). Если хотите перелинковку — Excel можно
загрузить вручную после префилла.

### 3.3. Изменений в `.env` для этого релиза НЕТ

Все правки — на уровне кода и UI. `diff` (см. §1.2) для этого обновления
ничего не покажет. Достаточно выполнить `git pull` + пересобрать контейнеры
(см. §2.4).

## Приложение A. Структура `.env` (актуальный список ключей)

Полный актуальный список с комментариями см. в `.env.example`. Ниже —
краткая карта по группам:

| Группа | Ключи | Где используются |
|---|---|---|
| Database | `POSTGRES_*`, `DATABASE_URL` | `backend/src/db.js`, миграции |
| Auth / API | `JWT_SECRET`, `BACKEND_PORT` | `backend/src/middleware/auth.js`, `server.js` |
| LLM keys | `GEMINI_API_KEY`, `DEEPSEEK_API_KEY`, `XAI_API_KEY` | `backend/src/services/llm/*.adapter.js` |
| LLM models | `GEMINI_MODEL`, `DEEPSEEK_MODEL`, `XAI_MODEL` | те же адаптеры |
| Прокси | `GEMINI_PROXY_*` (или `LLM_PROXY_*` / `HTTPS_PROXY` как fallback) | gemini/grok адаптеры (прокси обязательный для GEMINI и Grok из РФ) |
| Concurrency | `DEEPSEEK_MAX_CONCURRENT=8`, `GEMINI_MAX_CONCURRENT=6`, `XAI_MAX_CONCURRENT=4` | `services/llm/rateLimiter.js` |
| Качество | `LSI_COVERAGE_TARGET=80`, `EEAT_PQ_TARGET=7.5`, `INFO_ARTICLE_LSI_TARGET`, `INFO_ARTICLE_EEAT_TARGET` | `utils/objectiveMetrics.js`, info/link-article pipeline |
| Кеш Gemini | `GEMINI_CONTEXT_CACHE_ENABLED`, `INFO_ARTICLE_GEMINI_CACHE_ENABLED`, `LINK_ARTICLE_GEMINI_CACHE_ENABLED` | соответствующие пайплайны |
| Опциональный судья | `STAGE8_EVALUATOR_ENABLED=false`, `STAGE8_TEXT_CAP=12000` | `services/pipeline/stage8.js` |

## Приложение B. Что точно НЕ ломается между релизами

Эти инварианты намеренно поддерживаются — можете на них полагаться:

* `.env` живёт только локально, в `.gitignore`, **никогда** не приезжает с `git pull`.
* Миграции в `migrations/*.sql` — **идемпотентны и additive only** (никаких `DROP`).
  Откат кода без отката схемы безопасен.
* Все новые env-переменные — **опциональны**, у каждой есть рабочий default
  в коде. Минимально рабочий `.env` со старого релиза продолжит работать
  после `git pull` без правок.
* `docker-compose.yml` подключает `.env` через `env_file: .env`. Если файла
  нет — backend упадёт с явным сообщением «Missing GEMINI_API_KEY», а не молча.
